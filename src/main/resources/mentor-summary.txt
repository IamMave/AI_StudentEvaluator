Key Topics Covered

TestNG purpose and benefits

TestNG is a testing framework (agnostic to Selenium/Appium/Playwright/API) used to structure test suites, group tests, and control execution flow (suites, tests, classes, methods).

Distinction highlighted between tools (e.g., Selenium, Appium) and frameworks (the reusable, agnostic automation framework built around tools).

TestNG annotations and hooks (execution flow)

The main hooks discussed: @BeforeSuite, @AfterSuite, @BeforeTest, @AfterTest, @BeforeClass, @AfterClass, @BeforeMethod, @AfterMethod, and @Test.

Execution order explained: suite → test run → class → before-class → before-method → test method → after-method → after-class → after-test → after-suite (examples demonstrated).

Emphasis on using these hooks to run setup/teardown tasks (e.g., login once per class vs per method).

testng.xml (structure and usage)

Structure: contains runs, which list classes and their methods; parameters can be passed to control environment (e.g., browser choice).

Practical uses: define suites for sanity/regression, run tests in parallel/sequentially, control test selection and reporting.

Ways to trigger tests: run class directly, right-click testng.xml, Maven CLI, CI/CD pipelines.

Related automation topics and interviews

Waits: explicit vs fluent vs implicit — fluent wait discussed as resource-optimizing and suitable when running many tests or on limited resource device farms; explicit waits are commonly used.

BDD vs TDD and tools:

Cucumber (BDD) explained as using Given/When/Then; useful when requirements are flaky but less impactful in many orgs.

TDD defined as shift-left where tests drive development; different from BDD.

Page Object Model (POM) clarified as independent of TestNG/Cucumber; POM is a design pattern to organize UI interactions.

TestNG usable across API automation, mobile, and web tests.

Questions, Clarifications & FAQ Highlights

Can TestNG be used with Appium/Playwright/API tests? — Yes; TestNG is independent of the underlying tool.

What is the mapping between testng.xml run names (e.g., "Chrome execution") and actual driver behavior? — testng.xml needs parameters that are read in setup (e.g., @BeforeMethod) to initialize the correct driver/environment.

If a method lacks TestNG annotations (or is a plain main method), will TestNG execute it? — No: when run via testng.xml, TestNG executes annotated test methods; main() will execute only if the class is run directly (IDE run), not via testng.xml.

Do tests continue after failure? — Yes: TestNG will continue executing remaining tests unless explicitly configured otherwise; this was demonstrated by intentionally failing a locator and observing remaining tests run.

How to group or run subsets (sanity/regression)? — Create separate suites/testng.xml files or separate blocks with the required classes/methods; commonly a dedicated testng.xml is used for frequent suites (sanity) run by pipelines.
